# F1.55 — Curriculum Factory

**Doel:** niet 700–1000 leerdoelen met de hand schrijven, maar de **machine** bouwen die ze produceert. Elke syllabuswijziging of nieuw vak wordt dan "factory draaien", niet "opnieuw schrijven".
**Status:** kern gebouwd en bewezen op 2 vakken (bi, sk) · **Datum:** 2026-07-22 · geen deploy (offline tooling).

---

## 1. De pijplijn

```
Bronnen → Extractor → Briefing → Draft → Validate → Ingest → Tagger → Coverage → Dashboard → Review → Approved
```

| Stap | Wat | Geautomatiseerd? | Bestand |
|---|---|---|---|
| **Bronnen** | Slagio-content (begrippen, samenvattingen, vragen) + *optioneel* extern (CvTE/syllabus/examenblad) | intern: ja · extern: pluggable | `data-*.js`, `sam-*.js` |
| **Extractor + Briefing** | concepten (met belang-frequentie), sam-thema, misconceptie-signalen, schema, coverage-doelen | **ja** | `curriculum-factory.js brief` |
| **Draft** | concepten → 4–8 leerdoelen per domein volgens schema | **pluggable** (nu mens/agent, later API) | `factory/draft-*.json` |
| **Validate** | schema-check: enums, id-format, 4–8/domein, concepten ⊆ begrippen | **ja** | `curriculum-factory.js validate` |
| **Ingest** | leerdoelen → `knowledge-<niveau>.js` mét provenance | ja (volgende bouwstap) | — |
| **Tagger + Coverage** | vragen ↔ leerdoelen met confidence/source | **ja** (bestaat) | `tag-leerdoelen.js`, `leerdoel-dekking.js` |
| **Dashboard + Review** | syllabus/AI-Ready/review per leerdoel | **ja** (bestaat) | `curriculum-dash.js` |

De factory automatiseert **alles behalve de creatieve draft-stap**. Dat verandert het werk van *"schrijf 700 leerdoelen"* naar *"review 700 auto-gescaffolde leerdoelen"*.

## 2. Pluggable stages (bewust losgekoppeld)

- **Crawler (externe bronnen).** In deze omgeving **niet bereikbaar** (examenblad.nl/cvte.nl → netwerk geblokkeerd). De factory draait daarom volledig op interne content; de crawler is een extra bronlaag die aangaat zodra het netwerk/bron beschikbaar is. `externeBronnen.status` in de briefing markeert dit expliciet.
- **Draft-engine.** In deze omgeving **geen `ANTHROPIC_API_KEY`** → een script kan de API niet autonoom aanroepen. De draft-stap is nu **agent/mens-in-de-lus** (ik vul de briefing volgens het schema). De interface is API-klaar: dezelfde briefing is straks de prompt-input voor een automatische Claude-call (met provenance: model, prompt, datum, score).

> Dit is een eerlijke inperking, geen ontwerpfout: de *machine eromheen* (extractie, schema, coverage, validatie, provenance, dashboard) is volledig deterministisch en herbruikbaar. Alleen de creatieve kern wisselt van motor (mens → API) zonder dat de rest verandert.

## 3. Wat de Extractor per domein oplevert

`node scripts/curriculum-factory.js brief havo sk` → `factory/brief-havo-sk.json`:

- **concepten** — alle begrippen, gesorteerd op **frequentie in vragen** (belang-signaal). Bijv. Scheikunde C: `Zuur·31, Base·23, pH·17…`.
- **samThema** — excerpt uit de samenvatting (thematische context).
- **misconceptieSignalen** — zinnen uit de uitleg die een verwarring benoemen ("verwart X met Y", "in tegenstelling tot…").
- **schema + opdracht + richtlijn** (4–8 leerdoelen, enums voor vaardigheid/examenskill/relevantie).

## 4. Bewezen + eerlijke bevindingen

- **Generaliseert:** getest op bi (25 begrippen/domein-niveau) én sk (88 concepten, 5 domeinen). De frequentie-ranking legt vanzelf de kernconcepten bovenaan.
- **sam-context werkt:** 5/5 sk-domeinen, 4/4 bi-domeinen leveren een thema-excerpt.
- **Misconcepties zijn de zwakke plek:** auto-extractie uit uitleg is vak-afhankelijk — bi geeft 7 signalen, sk maar 1 (dunne uitleg). **Dit is precies waar de LLM/menselijke review de meeste waarde toevoegt**; de factory levert hints, geen eindproduct. Rijkere misconcepties komen later uit `vraag_fout`-events (F4) en/of correctievoorschriften (crawler).

## 5. Provenance (vanaf de bron)

Elk gegenereerd leerdoel krijgt bij ingest een `_herkomst`: `{ factory, model/agent, datum, bronnen, briefing-hash }`. Zo is straks per leerdoel te zien waaruit en waarmee het is gemaakt — de "bewaar bron/prompt/model/datum"-eis uit de Execution Brief.

## 6. Ingest-engine (gebouwd — de laag die jaren meegaat)

De volledige pijplijn `brief → draft → validate → ingest → tag → dashboard` draait. De **draft-stap is het enige vervangbare onderdeel**; alle kwaliteitscontroles blijven gelijk ongeacht welke motor (mens/LLM) draft.

- **Source-of-truth per vak:** `knowledge/<niveau>/<vak>.json`. Een **assembler** (`curriculum-factory.js assemble`) genereert daaruit `knowledge-<niveau>.js` — deterministisch (2× = byte-identiek).
- **`ingest`** (`curriculum-factory.js ingest <niveau> <vak> [--dry-run]`):
  - valideert eerst (schema/enums/id/integriteit; breekt af bij een issue);
  - **idempotent** — dezelfde draft twee keer geeft exact hetzelfde bestand;
  - **versiebeheer per leerdoel** — `_meta.version` bumpt alleen bij een échte inhoudswijziging (content-hash); `reviewStatus` valt dan terug naar `draft`;
  - **stabiele identifiers** — ID is de sleutel; titel/beschrijving mogen vrij evolueren zonder ID-wijziging;
  - **provenance** — `_meta: {version, reviewStatus:draft|reviewed|approved, lastReviewed, source, created, updated, contentHash}`;
  - **diff-rapport vóór schrijven** — toegevoegd / gewijzigd (v→v) / ongewijzigd / behouden;
  - behoudt bestaande leerdoelen die niet in de draft staan (verwijdert nooit stil).

`smoke.mjs` bewaakt: knowledge-havo.js in sync met de JSON-bronnen, geldige provenance, sidecar ⊆ bron voor **alle** getagde vakken.

## 7. Tweede pilot: Scheikunde (end-to-end bewezen)

De volledige loop op een niet-biologie-vak: `brief` (88 concepten) → draft (28 leerdoelen, 5 domeinen) → `validate` (schema-ok) → `ingest` (28 toegevoegd, daarna idempotent; 1 echte edit → v1→v2) → `tag` (432 vragen, 0.90 confidence, 401 concept_match). De koppelsleutel werd **domein-gescoped** nadat sk exact-dubbele begripvragen ("Mengsel" in domein A én B) blootlegde. In het dashboard verschijnt Scheikunde automatisch als **🔴 Concept** (draft) met de onzekere leerdoelen gemarkeerd — de kwaliteitscontrole doet zijn werk. Bewijst: het systeem is niet biologie-specifiek.

## 8. Roadmap vanaf hier

- **F1.6 Alle vakken** — per vak dezelfde loop; batchgewijs, kwaliteitslat gelijk. Scheikunde-draft wacht nu op menselijke review (→ `reviewStatus: reviewed/approved`).
- **F1.7 Review** — menselijke sign-off + eventuele overrides voor de fallback-vragen.
- **Draft-motor upgraden** — zodra een `ANTHROPIC_API_KEY` beschikbaar is, vervangt een Claude-call de handmatige draft-stap; de rest van de pijplijn verandert niet.
- **F2+** — Question Intelligence op een complete, gevalideerde curriculumlaag.

**Geen nieuwe dashboards** — focus op data/curriculumlaag, zoals afgesproken.
